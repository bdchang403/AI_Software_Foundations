# AI Model Governance: The AI-SBOM

## 🧩 What is an SBOM and Why is it Crucial?

A **Software Bill of Materials (SBOM)** is a formal, machine-readable inventory of all software components, libraries, and dependencies used in a system. Just as a food label lists ingredients to protect consumers from allergens, an SBOM protects organizations from security and supply-chain vulnerabilities.

### The Regulatory Lineage (Citations)
The concept of the **AI-SBOM** (or AI-BOM) is an evolution of established cybersecurity standards:
*   **Executive Order 14028 (2021):** The U.S. government mandated SBOMs for all federal software suppliers to improve supply chain transparency.[1]
*   **NTIA/CISA Standards:** The National Telecommunications and Information Administration (NTIA) and the Cybersecurity and Infrastructure Security Agency (CISA) defined the "Minimum Elements" for SBOMs.[2]
*   **SPDX 3.0:** The industry-standard format for SBOMs was updated in 2024 to include specific profiles for **AI Models and Datasets**, allowing them to be tracked alongside traditional code.[3]
*   **EU AI Act & NIST AI RMF:** These global regulatory frameworks now emphasize **Traceability** and **Provenance** as mandatory requirements for high-risk AI systems.

[1] [EO 14028: Improving the Nation’s Cybersecurity](https://www.whitehouse.gov/briefing-room/presidential-actions/2021/05/12/executive-order-on-improving-the-nations-cybersecurity/)
[2] [NTIA: Minimum Elements for a Software Bill of Materials](https://www.ntia.gov/report/2021/minimum-elements-software-bill-materials-sbom)
[3] [SPDX 3.0: AI Profile Specification](https://spdx.dev/)

**Why it's crucial for AI-Augmented Teams:**
In the era of AI-generated code, a standard SBOM is no longer enough. If an AI model is later discovered to have a security flaw, a training data bias, or a licensing violation, you must be able to identify exactly which lines of code were influenced by that model.

Without an **AI-SBOM**, your codebase becomes an unauditable "Black Box." You lose the ability to:
1. **Patch at Scale:** If Model X is found to have a security flaw, you can't find the code it wrote.
2. **Prove Provenance:** You cannot legally prove that your code was not generated by a model trained on unlicensed data.
3. **Debug Drift:** You won't know if a sudden production bug is linked to a silent "Model Update" from your AI provider.

## 1. The AI-SBOM (Software Bill of Materials)

Standard SBOMs track library dependencies (e.g., `npm`, `pypi`). The **AI-SBOM** tracks the "Cognitive Dependencies." Every pull request generated by an AI Agent must include a metadata header:

```yaml
# AI-SBOM Header
Agent: "Auth-Agent-01"
Model: "gemini-1.5-pro-002"
Temperature: 0.0 (Deterministic)
Context_Nodes: ["EPIC-101", "ADR-042", "STORY-112"]
Verification_Timestamp: "2026-05-01T16:11:35Z"
```

## 2. Model Versioning & Drift

Regulated organizations must treat Model Updates (e.g., GPT-4o to GPT-4o-mini) as a **Major Change**. 
*   **Drift Analysis:** Before upgrading a production agent's model, the BuilderOps team must run the "Adversarial Test Suite" against the new model to ensure its logic hasn't "drifted" from the original constitutional constraints.
*   **Versioning Policy:** Always pin your agents to a specific model version. Never use the `latest` tag in a regulated production environment.

## 3. Data Sovereignty & Residency

The AI-SBOM must verify that the training data and inference traffic comply with local laws (e.g., GDPR, CCPA).
*   **Regional Inference:** Agents processing EU citizen data should utilize inference endpoints located within the EU.
*   **Data Minimization Audit:** The Security Lead must verify that the "Context Chaining" logic does not inadvertently leak PII into the inference stream.
